## Цель работы

Цель лабораторной работы — изучить расширенные возможности Git: работу с локальным и удалённым репозиторием, ветками, конфликтами слияния, Git hooks и моделью ветвления Git Flow.

В ходе работы были выполнены следующие действия:

- создан и склонирован удалённый репозиторий;

- выполнена работа с ветками и удалённым репозиторием;

- настроен `pre-commit` hook для проверки `.txt` файлов;

## 8. Настройка `pre-commit` hook

Для автоматизации проверки формата файлов был создан скрипт `check_format.sh`.

В рамках работы была задана простая спецификация формата `.txt` файлов:

1. Проверяются только файлы с расширением `.txt`.
2. Файл не должен быть пустым.
3. В конце строк не должно быть лишних пробелов или табуляций.

Скрипт использует `grep`, потому что эта утилита позволяет искать строки по шаблону. Например:

```bash
grep -n '[[:blank:]]$' "$file"
```

Эта команда ищет строки, которые заканчиваются пробелом или табуляцией.

Скрипт был скопирован в папку `.git/hooks` под именем `pre-commit`, после чего ему были выданы права на выполнение:

```bash
cp check_format.sh .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
ls -l .git/hooks/pre-commit
```

![Копирование check_format.sh в .git/hooks/pre-commit и выдача прав на выполнение](./images/screen_41.png)

*Рисунок 41. Копирование check_format.sh в .git/hooks/pre-commit и выдача прав на выполнение.*

В выводе команды `ls -l` видны права:

```text
-rwxr-xr-x
```

Буква `x` означает, что скрипт является исполняемым.

Для проверки работы hook был создан файл `bad_format.txt` с ошибкой форматирования — пробелами в конце строки. При попытке коммита hook заблокировал операцию:

```bash
git add bad_format.txt
git commit -m "Проверка pre-commit hook"
```

![Pre-commit hook заблокировал коммит из-за ошибки форматирования](./images/screen_42.png)

*Рисунок 42. Pre-commit hook заблокировал коммит из-за ошибки форматирования.*

После этого сам скрипт `check_format.sh` был добавлен в репозиторий и отправлен на GitHub:

```bash

git commit -m "Добавлен скрипт проверки формата"
git push origin main
```

## 12. Создание hotfix-ветки `v1.0.1`

После релиза была смоделирована ситуация обнаружения критической ошибки. Ошибка заключалась в том, что функция создания задачи позволяла создать задачу без названия.

Для исправления была создана hotfix-ветка:

```bash
git checkout main
git flow hotfix start v1.0.1
```

В файл `task_manager.py` была добавлена проверка пустого названия задачи:

```python
def create_task(title, description):
    if not title:
        raise ValueError("Название задачи не может быть пустым")

    # Логика создания задачи
    print(f"Создана новая задача: {title}")
```

Изменение было закоммичено:

```bash
git add task_manager.py
git commit -m "Исправлена критическая ошибка создания задачи"
```

При завершении hotfix у установленной версии `git-flow` возникла такая же проблема с автоматическим созданием тега. Поэтому hotfix был завершён вручную:

```bash
git tag -a v1.0.1 -m "Release v1.0.1"
git checkout develop
git merge --no-ff hotfix/v1.0.1 -m "Merge hotfix v1.0.1 into develop"
git branch -d hotfix/v1.0.1
```

После проверки было видно, что:

- тег `v1.0.1` создан;
- временная ветка `hotfix/v1.0.1` удалена;
- исправление попало в `main`;
- исправление было возвращено в `develop`.
